## 1. 项目背景与动机

### 1.1 现状问题

现有 Zotero AI 插件在处理 PDF 时存在以下核心问题：

| 插件 | PDF处理方式 | 问题 |
|------|------------|------|
| Beaver | 直接提取文本层，OCR 回退 | 扫描件质量差，无结构化输出 |
| PapersGPT | 全文提取 + 本地 RAG | 无 OCR，纯文本丢失格式 |
| Aria (zotero-gpt) | 文本提取 | 无 RAG，无 OCR |
| Zotero-PDF-Translate | 文本提取用于翻译 | 翻译专用，非通用 AI |
| Better Notes | 标注提取 + 笔记 | 无 AI 能力 |

**核心痛点：**
- PDF 文本层提取丢失结构（标题、段落、列表层级）
- 扫描件/图片 PDF 完全无法处理
- 公式、表格转为纯文本后不可读
- 空格、异体字、断字问题浪费 AI token 且影响理解
- **没有插件将 PDF→MD 转换 + 标注映射结合起来**

### 1.2 解决思路

将 PDF 通过高质量 OCR/文档理解模型转换为结构化 Markdown：
- Markdown 保留标题层级、列表、表格、公式（LaTeX）
- 扫描件通过 VLM 视觉理解，不依赖文本层
- 标注通过模糊文本匹配映射到 MD 对应位置
- AI 直接消费 Markdown，token 利用率和理解质量大幅提升

### 2.2 核心接口设计

#### IOcrBackend 接口

```typescript
interface IOcrBackend {
  readonly id: string;           // 唯一标识, 如 "glm-ocr"
  readonly displayName: string;  // UI 显示名, 如 "智谱 GLM-OCR"
  readonly isCloud: boolean;     // 是否需要网络

  validateConfig(config: BackendConfig): Promise<{ valid: boolean; error?: string }>;

  convert(
    pdfPath: string,
    config: BackendConfig,
    onProgress?: ProgressCallback,
  ): Promise<ConversionResult>;
}
```

#### ConversionResult

```typescript
interface ConversionResult {
  markdown: string;        // 完整 MD 文本
  pages: PageResult[];     // 按页结果（用于标注映射）
  metadata?: {
    backend: string;
    duration: number;
    pageCount: number;
    tokensUsed?: number;
  };
}

interface PageResult {
  pageIndex: number;       // 0-based 页码
  markdown: string;        // 该页的 MD 内容
  images?: ImageResult[];  // 提取的图片
}
```

#### BackendConfig

```typescript
interface BackendConfig {
  apiKey?: string;
  apiEndpoint?: string;
  modelName?: string;
  localServerUrl?: string;
  localBinaryPath?: string;
  maxConcurrentPages?: number;
  timeout?: number;
}
```

### 3.2 推荐优先实现顺序

1. **GLM-OCR (智谱)** — 性价比最高，中文优秀，云API简单
2. **通用 VLM 后端** — 覆盖所有 OpenAI 兼容端点（含 DeepSeek、Qwen、本地 Ollama 等）
3. **MinerU** — 最佳免费本地方案
4. **Mistral OCR 3** — 国际方案中表格/公式最强

### 3.3 各后端实现要点

#### GLM-OCR

```
端点: POST https://open.bigmodel.cn/api/paas/v4/layout_parsing
认证: Authorization: Bearer <API_KEY>
输入: PDF 文件 Base64 data URI (data:application/pdf;base64,...)
输出: JSON 含 md_results 字段（Markdown 文本）+ layout_details 数组
参数:
  - model: "glm-ocr"
  - file: base64 data URI
  - return_crop_images: true  → 返回裁剪图片 URL（layout_details 中 label="image" 的 content 字段）
注意:
  - md_results 中图片显示为占位符: ![](page=N,bbox=[x1,y1,x2,y2])
  - 需用 layout_details 中的图片 URL 替换占位符，或下载后转 base64 嵌入
  - 图片 URL 过期时间未知（官方未文档化），建议转换时立即下载
```

#### 通用 VLM 后端（最灵活）

```
端点: 用户自定义 (如 http://localhost:11434/v1/chat/completions)
认证: 用户提供的 API Key
输入: 每页渲染为 PNG → Base64 → 作为 image_url 发送
输出: chat completion 中的 markdown 内容
兼容: OpenAI / Ollama / vLLM / LiteLLM 等所有兼容端点
```

**关键**: 需要 PDF→图片渲染工具，使用 Zotero 内置的 pdf.js。

#### MinerU 本地

```
模式1: CLI — 调用 magic-pdf 命令行
模式2: HTTP — POST http://localhost:{port}/pdf/parse
输入: PDF 文件路径
输出: 生成 .md 文件在输出目录
```

## 4. 标注映射方案

### 4.1 问题定义

PDF 标注有精确的页面坐标和文本内容，但转换后的 MD 文本与原始 PDF 文本不完全一致（OCR 误差、格式化差异、断字处理等）。需要在 MD 中找到标注对应的位置。

### 4.2 算法设计

```
输入:
  annotations[]  — Zotero 中的 PDF 标注（含 text, pageIndex, comment, color）
  conversionResult — 转换结果（含 pages[] 和完整 markdown）

对每个 highlight/note 类型标注:
  1. 获取标注所在页码 pageIndex
  2. 获取对应页的 MD 文本 pages[pageIndex].markdown
  3. 文本预处理:
     - 统一空白字符
     - 移除断字连字符
     - Unicode 规范化
  4. 尝试精确子串匹配（快速路径）
  5. 若失败，滑动窗口模糊匹配:
     a. 窗口大小 = 标注文本长度 ± 20%
     b. 步长 = max(1, 标注长度/10)
     c. 计算 Levenshtein 相似度
     d. 取最高分且 >= 阈值(0.7) 的匹配
  6. 若仍失败，扩展到相邻页搜索
  7. 计算在完整 MD 中的绝对偏移量
```

### 4.3 标注插入格式

使用 HTML 注释，不影响 MD 渲染：

```markdown
这是一段文字，其中 <!-- ann:highlight color="#ffff00" id="ABC123" -->被高亮的部分<!-- /ann --> 继续。

<!-- ann:note color="#ff6666" id="DEF456" comment="这是用户的笔记内容" -->
对应的标注段落文字
<!-- /ann -->
```

**插入顺序**: 从后往前（高偏移量优先），避免偏移量漂移。

## 5. 存储方案

### 5.1 设计决策

**核心原则：插件只做 PDF → MD，保持 Markdown 格式，不转 HTML。**

讨论过两种方案：
- ❌ 存为 Zotero 笔记（HTML）：偏离 pdf2md 定位，且 Zotero 笔记底层是 HTML，需要 MD→HTML 转换
- ✅ 存为 Zotero .md 附件：保持 MD 格式，出现在条目树中，双击用外部编辑器打开

### 5.2 Better Notes 兼容

Better Notes（windingwind/zotero-better-notes）使用 Zotero 原生笔记系统（HTML），提供 MD ↔ HTML 双向转换。

- **无 BN**：.md 附件正常可用，双击用外部编辑器（VS Code、Typora 等）打开
- **有 BN**：可选检测 `Zotero.BetterNotes.api.import.fromMD()` 额外创建可编辑笔记
- BN 不是依赖，只是增强。不装也完全能用，装了锦上添花

### 5.3 Zotero 附件存储

```typescript
// 转换后的 MD 存储为 Zotero stored file attachment
await Zotero.Attachments.importFromFile({
  file: tmpPath,           // 临时文件路径
  parentItemID: parentID,  // 父条目 ID
  title: "paper_name.md",
  contentType: "text/markdown",
});

// 添加标签标记为自动生成
attachment.addTag("pdf2md:converted", 1);
```

### 5.4 重复处理

转换前检查是否已有 `pdf2md:converted` 标签的 MD 附件：
- 已存在 → 提示：覆盖 / 跳过 / 新建
- 不存在 → 直接创建

## 9. 待讨论问题

### 9.1 向量化需求
转换为 MD 后是否需要内置向量化？
- **方案 A**: 不内置，由其他插件（PapersGPT 等）消费 MD 文件
- **方案 B**: 内置简单的分块索引，方便 AI 检索
- **当前倾向**: 方案 A，专注做好转换和标注映射，不重复造轮子

### 9.2 图片处理 ✅ 已决定

PDF 中的图表处理方案（经调研业界做法后确定）：

**当前问题**: GLM-OCR 默认返回 `![](page=N,bbox=[x1,y1,x2,y2])` 占位符，图片实际丢失。

**业界做法对比**:
| 工具 | 方案 |
|------|------|
| MinerU | 单独文件存 images/ 目录 |
| Marker | base64 (JSON 输出) + 可选 LLM 描述 |
| Mathpix | 云端 CDN URL（30天过期） |
| TextIn (合合) | 云端 URL（30天过期）或 base64 可选 |
| Docling (IBM) | 三种模式：占位符 / base64 / 单独文件 + VLM 描述 |
| Mistral OCR | base64（`include_image_base64=true`） |

**决定**:
- **默认方案 (Phase 1)**: 开启 `return_crop_images: true`，拿到图片 URL 后立即下载转 base64 嵌入 MD。自包含，不怕 URL 过期
- **后续优化 (Phase 2)**: 图床上传选项（SM.MS / GitHub / 自建），永久 URL 且文件不臃肿
- **后续优化 (Phase 3)**: VLM 描述模式，将图片转为自然语言描述（对 AI 最友好）
- 在偏好设置中让用户选择图片处理方式

### 9.3 大文件处理
超过 100 页的 PDF 如何处理？
- 分批处理（每批 20 页）
- 显示预估时间和成本
- 支持中断和续传
